# End-to-End Project Workflow

## 📝 Layman Explanation (In Brief)
Think of this system as an **Automated Security Guard** for your crypto investments. When you ask about a token, the engine performs a "Digital Background Check":
1. **Identifies the Asset**: Confirms the exact token and its official contract address.
2. **Gathers Evidence**: Scans the blockchain and official databases for "red flags" (like high taxes, hidden traps, or regulatory warnings).
3. **Verifies Safety**: Checks if the token's code is open and if it's been audited by security experts.
4. **Reports Facts**: Provides a concise summary of the risks found. If you need more depth, it offers a "Detailed Risk Profile" with easy-to-read **Green/Yellow/Red** safety badges.

---

This document explains the technical execution pipeline of the **Crypto Risk Verification Assistant**, from the moment a user enters a query to the final display of the risk profile.

## 1. User Input & Frontend (React)
- **Action**: User enters a token name (e.g., "Uniswap"), ticker ("UNI"), or contract address in the search bar.
- **Trigger**: The `handleVerify` function in `App.tsx` captures the query and calls the backend API.

## 2. API Entry & Intent Routing (FastAPI)
- **Endpoint**: `POST /api/v1/chat`
- **Query Router**: Before processing, the `QueryRouter` scans for "investment advice" or "out-of-scope" intents. 
  - *If Advisory*: Returns a static refusal (e.g., "I cannot provide investment advice").
  - *If Factual*: Pipeline continues.

## 3. Identity & Asset Classification
- **Entity Resolution**: The `EntityResolver` calls the **CoinGecko API** to map the user's input to a canonical `asset_id`. It resolves the official contract address, chain, and metadata.
- **Asset Classifier**: The system determines the asset "type":
  - `smart_contract_token`: EVM-based tokens with contract addresses (triggers full security scans).
  - `native_asset`: Base coins like BTC, XRP, SOL (scans market/regulatory data only).
  - `low_data_asset`: Assets with minimal market presence.

## 4. Multi-Layer Signal Engine
The system concurrently gathers factual data from independent sources:
- **Market Data**: Fetched from CoinGecko (Market Cap, Rank, Exchange Listings).
- **Security Signals**: Fetched via **GoPlus Security API**. This extracts critical risk flags:
  - Honeypot status
  - Buy/Sell Tax percentages
  - Proxy/Upgradable contract detection
  - Minting permissions
  - Verified source code status
- **Audit Detection**: The `AuditDetector` cross-references known providers (CertiK, Hacken) in the asset's metadata and official links.

## 5. Signal Normalization & RAG
- **Normalization**: The `SignalNormalizer` converts raw numbers into risk indicators:
  - *Example*: A buy tax of 6% is converted to `⚠️ PREDATORY TAX (>5% threshold)`.
  - *Example*: A missing audit is explicitly labeled as `No audit found`.
- **RAG Retrieval**: The `RAGService` queries the **ChromaDB Vector Store** using the normalized signals to find relevant regulatory warnings (FCA, SEC) or explanatory context for the specific risks detected.

## 6. LLM Explanation (Groq / Llama 3)
- **Constraint**: The LLM is **not** allowed to generate new facts or predictions.
- **Prompting**: It is given the normalized signals and retrieved context. It is instructed to provide a **2-3 line factual summary** explaining the detected risks in neutral, non-advisory language.

## 7. Output Formatting & Post-Guards
- **Formatting**: The `OutputFormatter` builds the final Markdown response with strict headers:
  - `Asset Profile`
  - `Smart Contract & Protocol Risk`
  - `Market & Liquidity Risk`
  - `External Trust & Verification`
  - `Regulatory Sanctions & Alerts`
- **Validation**: `PostGuards` runs a final regex check to ensure no "speculative" or "advisory" words (like "buy", "suggests", "potential") were generated by the LLM.

## 8. Final UI Display
The React frontend receives a JSON object containing:
1. **The Markdown Response**: Rendered in the chat bubble.
2. **The Raw Risk Profile**: If the asset has security data, a **"🔍 View Detailed Risk Profile"** button appears.
3. **Expandable Accordion**: Clicking the button reveals the **Risk Accordion**, which uses a color-coded schema (**Green**, **Yellow**, **Red**) to show the depth of the security analysis (TokenSniffer-style).
